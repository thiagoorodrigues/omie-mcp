# Omie MCP v0.1 — Design

**Data:** 2026-07-20
**Status:** Aprovado

## Objetivo

Servidor MCP (stdio, TypeScript/Node) para a API da Omie
(https://developer.omie.com.br/service-list/), permitindo, a partir do
Claude, **listar produtos**, **listar clientes** e **criar orçamentos**,
mais tools de consulta de apoio. A arquitetura espelha o `runrun-mcp`
(`/Users/thiagorodrigues/Projetos/south/runrun-mcp`).

## Escopo v0.1

Tools:

| Tool           | Endpoint             | call Omie         | Descrição |
|----------------|----------------------|-------------------|-----------|
| `products_list`| `/geral/produtos/`   | `ListarProdutos`  | Paginação + filtros básicos |
| `products_get` | `/geral/produtos/`   | `ConsultarProduto`| Por `codigo_produto` |
| `clients_list` | `/geral/clientes/`   | `ListarClientes`  | Paginação + filtro por nome/cnpj (via `clientesFiltro`) |
| `clients_get`  | `/geral/clientes/`   | `ConsultarCliente`| Por `codigo_cliente_omie` |
| `budget_create`| `/produtos/pedido/`  | `IncluirPedido`   | Simplificado: `codigo_cliente` + `itens[]` |
| `budgets_list` | `/produtos/pedido/`  | `ListarPedidos`   | Paginação + filtros de data |

Fora de escopo: editar/excluir produtos, clientes ou pedidos; faturamento;
NF-e; multi-empresa por chamada.

## Arquitetura

```
src/

  config.ts         # env vars + validação zod
  client.ts         # OmieClient — wrapper JSON-RPC da Omie
  errors.ts         # OmieApiError + helpers de resposta MCP
  pagination.ts     # page/limit -> pagina/registros_por_pagina

```

## Config (`config.ts`)

Env vars (validadas com zod):

- `OMIE_APP_KEY` — obrigatória
- `OMIE_APP_SECRET` — obrigatória
- `OMIE_BASE_URL` — default `https://app.omie.com.br/api/v1`
- `LOG_LEVEL` — default `info` (`debug|info|warn|error`)

## Client (`client.ts`) — principal diferença do runrun

A Omie não é REST; é um POST único por endpoint no padrão JSON-RPC. API:

```ts
async call<T>(endpoint: string, callName: string, param: object): Promise<T>
```

Monta o body:

```json
{ "call": "<callName>", "app_key": "...", "app_secret": "...", "param": [ <param> ] }
```

Faz `POST` para `baseUrl + endpoint`. O `parseResponse` trata o envelope
de erro da Omie: ela retorna `faultstring`/`faultcode` (frequentemente com
HTTP 500, às vezes em corpo de resposta "ok"). Se `faultstring` estiver
presente, lança `OmieApiError`.

## Paginação (`pagination.ts`)

Campos MCP `page`/`limit` mapeados para os campos Omie
`pagina`/`registros_por_pagina`. Defaults: page 1, limit 50 (máx 100).

## Detalhe das tools

- **products_list** — param `{ pagina, registros_por_pagina, ...filtros }`.
- **products_get** — param `{ codigo_produto }`.
- **clients_list** — param `{ pagina, registros_por_pagina, clientesFiltro: { nome?, cnpj_cpf? } }`.
- **clients_get** — param `{ codigo_cliente_omie }`.
- **budget_create** (simplificado) — input: `codigo_cliente` (number) +
  `itens: [{ codigo_produto, quantidade, valor_unitario }]` + `etapa`
  opcional (default = etapa de orçamento). A tool monta internamente o
  `cabecalho` (com `codigo_cliente`, `etapa`, `codigo_pedido_integracao`
  opcional) e o array `det` a partir dos itens, e chama `IncluirPedido`.
- **budgets_list** — param `{ pagina, registros_por_pagina, ...filtros de data }`.

Os nomes exatos dos campos de cada `param`/filtro serão confirmados contra a
documentação da Omie durante a fase de implementação.

## Erros (`errors.ts`)

`OmieApiError` guarda `status`, `faultcode`, `faultstring` e endpoint.
`successResponse` retorna JSON formatado; `genericErrorResponse` converte
erros de API/rede no shape de resposta MCP (`isError: true`). Rate limit da
Omie (ex.: consumo redundante) recebe nota amigável.

## Testes

Vitest, um arquivo por tool, usando um `mockClient` que mocka `call`.
Cobrem: montagem correta de `endpoint`/`callName`/`param`, defaults de
paginação, filtros repassados, montagem do payload do `budget_create`, e
`isError` em falha da API.
